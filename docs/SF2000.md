# SF2000 Technical Notes

<!-- SPDX-License-Identifier: MIT -->

These are the non-obvious details currently baked into the QEMU bring-up.

## Memory

The SF2000 has 128 MiB of RAM. The HCRTOS DTS used by the existing ports
defines:

```c
#define CONFIG_MEMORY_SIZE 0x08000000
```

QEMU maps that RAM at physical `0x00000000`. MIPS unmapped cached virtual
addresses at `0x80000000` therefore refer to the same RAM.

## CPU Clock

The physical SF2000 can run its MIPS core up to 918 MHz. QEMU uses 918 MHz as
the default CPU reference clock so CP0 count/timer behavior starts from the
maximum-frequency stock configuration. Lower-frequency experiments are still
useful when comparing firmware timing paths or power profiles; set
`SF2000_CPU_HZ=<hz>` in the environment to override the reference clock for a
run. Values outside `1 MHz..918 MHz` are rejected and the model falls back to
918 MHz.

## Boot Chain Inputs

There are two stock firmware inputs:

- SPI boot ROM:
  `/root/host-frogdev/universal/orig_firmware/SF2000_XMC_XM25QH40B_4mbit.bin`
- SF2000 application image:
  `/root/host-frogdev/universal/orig_firmware/bisrv_08_03.asd`

The SPI image has reverse-engineered symbols in:

```text
/root/host-frogdev/universal/orig_firmware/SF2000_XMC_XM25QH40B_4mbit.bin.ld
```

Important addresses from that file:

- `lcfg_header = 0x80f00000`
- `image_start = 0x80000200`
- `bl_start = 0x81000000`
- `bl_heap_start = 0x83000000`
- `bl_ramsize = 0x84000000`
- `CHIP_ID = 0xb8800000`

The ASD image starts with an `LCFG` header. The first 0x200 bytes are metadata,
then the image body is mapped as stock firmware data. The stock entry point is
at `0x80001000`.

After the bootloader jumps into `bisrv.asd`, stock maincode calls
`get_sysinfo_from_bl()` at `0x8026e79c`. The bootloader leaves a small handoff
descriptor in low RAM and advertises it through:

- `0xa0000010`: magic `0xabcd5aa5`;
- `0xa0000014`: kseg1 pointer to the handoff payload.

The pointer must be a CPU-visible uncached address, not a physical address.
QEMU seeds the payload at physical `0x00000100` and writes pointer
`0xa0000100`. The first payload word is currently `0x16`, the stock
`RGB_LCD` output route. Writing physical `0x100` into the pointer field makes
the firmware copy routine fault at `0x8028e70c` because it dereferences an
unmapped low virtual address.

Current QEMU supports two modes:

- `-bios <bootrom>`: loads the SPI boot ROM at physical `0x1fc00000`, matching
  the MIPS reset alias at virtual `0xbfc00000`.
- `-bios <bootrom> -kernel <bisrv.asd>`: direct bring-up mode. This loads the
  ASD at physical `0x00000000` and resets the CPU to virtual `0x80001000`.
- `make smoke-stock-full`: boots the stock bootloader with a generated raw SD
  image containing `/BIOS/bisrv.asd`.
- `make smoke-stock-full-bugfix`: boots the vendor bugfixed bootloader against
  the same generated FAT32 image and verifies handoff into the stock firmware.

The direct ASD path is still useful because it reaches the stock application
quickly. The full-chain target currently proves bootloader SD initialization
and raw sector reads from a generated FAT32 image. The stock bootloader reads
the MBR, FAT32 VBR, FSINFO sector, root directory, and `BIOS` directory, then
finds `BISRV.ASD`.

The original bootloader has a known file-object close bug around the directory
scan/open sequence. With minimal synthetic media it can reopen `BISRV.ASD`
through a stale directory pointer and derive compatibility LBA `0x202020` from
space-filled filename bytes. The vendor bugfixed bootloader redirects the
problem call at `0x81003b08` to a wrapper at `0x81020300`; with that image,
QEMU now reads the ASD, passes the bootloader CRC32/MPEG-2 check, enters stock
firmware display code, and mounts the FAT32 SD image.

## Address Translation

Stock firmware accesses registers through MIPS unmapped `kseg1`, for example
`0xb8800000`. QEMU sees those as physical addresses with the `0xa0000000`
offset removed:

```text
0xb8800000 virtual -> 0x18800000 physical
```

That is why the generic MMIO logger reports `0x188...` addresses even though
the firmware symbols use `0xb88...`.

## Chip ID

The first stock ASD block reads `CHIP_ID + 2` and branches on HC15xx values.
Returning zero makes the firmware take the wrong path. QEMU currently returns
`0x3503` for the halfword read at physical `0x18800002`, which matches one of
the known HC15xx branches in the stock startup code.

## Current Boot Status

`make smoke-stock-asd` confirms:

- the SPI image loads at `0x1fc00000`;
- the stock ASD loads at physical `0x00000000`;
- reset starts at `0x80001000`;
- early MMIO reaches the HC15xx chip ID and clock/pinmux setup block.

`make smoke-stock-fatfs` confirms that direct stock ASD execution also reaches
the SD/FatFs mount path. The SDIO model currently implements enough command
response plus DMA read/write behavior for firmware access to a generated card
or an optional raw `sd0` image. It is intentionally permissive: missing media
falls back to synthetic boot sectors so firmware probes keep advancing.

The SDIO DMA register at physical `0x1884c030` is modelled as a start/busy
latch plus completion bits. The stock bootloader writes `0x21` to start, waits
for bits `0x0c`, then acknowledges by writing back bit `0x40`.

Real-device hardware-probe logs captured initialized SDIO register words before
byte probing of `0xb884c00b` faulted on an early probe build:

- `0xb884c000`: `0x058de414`
- `0xb884c004`: `0xb3690000`
- `0xb884c008`: `0x05000200`

QEMU uses those values as 32-bit reset/default observations. Command-status and
transfer-status byte reads are still modelled behaviorally because stock
firmware polls those bytes during command execution and DMA completion.

The stock boot path issues `CMD55` followed by `CMD6 arg=2` after card
selection. QEMU treats that sequence as `ACMD6` and records a 4-bit SD bus
width in the trace. A bare `CMD6` is still accepted as a no-data switch command
until real traces show the firmware consuming the 64-byte SD switch-function
status payload.

The bugfixed stock bootloader path now reaches the stock firmware, mounts the
generated FAT32 image, and exercises GMA scanout. Direct ASD mode remains the
fastest path for display and filesystem iteration.

Implemented peripherals beyond raw boot include:

- UART0 and UART1 as minimal 16550-compatible ports. The HC15xx linker scripts
  place UART0 at `0xb8818300` and UART1 at `0xb8818600`, which QEMU sees as
  physical `0x18818300` and `0x18818600`. UART1 is the boot/debug console and
  is wired to interrupt bank 1 bit 17; UART0 currently shares the same
  transmit/status behavior without a routed interrupt because stock 08.03 only
  touches it during low-level setup paths observed so far.
- GPIO keypad scanning through the active-low L23/L24 shift-register contract.
  L23's output latch only triggers the parallel-load phase while the GPIO
  direction bit marks L23 as an output. After firmware changes L23 back to
  input, the retained low output latch must not reset the shift index on every
  L24 clock edge. This distinction is required for stock launcher navigation:
  QEMU monitor `sendkey` events now move the visible menu selection.
- Eight south timers with a synthetic scheduler tick on TIMER5/EIRQ3. TIMER0
  and TIMER1 are second/millisecond timers, not generic compare timers; their
  control reads expose current milliseconds in bits 16..25 and must not assert
  `INT_ST` for a zero target.
- SDIO command responses and DMA read/write transfers backed by `sd0`.
- GPIO-driven panel command/pixel writes for early display diagnostics.
- GPIO ISR write-one-to-clear behaviour for the stock power-detect path. GPIO R
  bit 30 is the power-good latch and bit 31 is the power-alert latch in the
  stock 08.03 firmware. Treating those acknowledgement writes as normal storage
  leaves power-alert latched and keeps the launcher from progressing.
- SAR ADC register defaults with a stable battery-range sample.
- Watchdog-disable acknowledgement so firmware can touch WDT without noise.

The south-bridge timer interrupt is edge-like. Before firmware programs TIMER5,
reading interrupt status bank 1 consumes the synthetic tick and schedules the
next one. Configured TIMER0/TIMER1/TIMER5 interrupts are routed through the same
bank-1 timer bit, but the CPU line is masked after the aggregate status read
until firmware acknowledges or reprograms a child timer. Leaving the line
level-asserted after that read re-enters the stock interrupt dispatcher and
lands in its nested-interrupt guard at `0x8033bf38`.

The SDIO completion interrupt shares the same external interrupt path but is
gated by interrupt-enable bank 1. Asserting it while disabled regresses the
bootloader into its interrupt path. With the enable gate in place, the vanilla
08.03 OS image reaches `[FS]successed!` after multi-sector reads from the FAT32
card image.

The next missing pieces are higher-fidelity GE/display timing, audio, USB, and
the rest of the low-power/standby peripheral behavior.

Set `SF2000_TRACE_PC=1` when running QEMU to print low-volume PC samples and
known stock-firmware landmarks to stderr and the QEMU log without enabling the
large `-d unimp` MMIO trace. This is useful once the SD image mounts, because
full tracing slows the firmware enough that long post-mount idle states are
hard to distinguish from slow progress. Landmark logging is off by default.

The trace landmarks also cover the stock launcher path (`run_task`,
`security_check`, `run_menu`, and core file calls). The vanilla OS-only SD
image now reaches `run_menu` without the diagnostic security bypass. The
important missing artifact was a factory/security area in SPI flash. Public
update bootloader images do not carry that personalized region, but the stock
08.03 launcher still reads it through `xmc_security_read()` before it creates
the menu task.

QEMU models the minimum observed contract for that factory region. During the
stock `xmc_security_read()` path, reads at flash `0x2000` provide neutral key
bytes, reads at `0x10c0` provide neutral additive bytes, and reads at `0x1000`
return check bytes derived from the runtime challenge string pointed to by the
firmware. The first eight check bytes are XORed with the key bytes stored by
`xmc_get_id()`, and later bytes follow the rolling XOR relation used by
`security_check()`. This is a behavioral model of the missing factory blob, not
a firmware patch.

The SPI-flash direct window also models the early identification commands used
by the stock bootloader: `0x9f` returns a JEDEC-style manufacturer/type/capacity
sequence, `0x05` reports a ready status byte, and `0xab` returns the electronic
signature after three dummy bytes. Other direct reads fall back to the loaded
boot ROM contents, except for the factory/security area described above.

For diagnosis only, `SF2000_PATCH_STOCK_SECURITY=1` patches the stock
`security_check` prologue in RAM after the bootloader hands off to the ASD.
This is intentionally not enabled by any smoke target and is not considered a
correct emulation model. It remains available as a comparison probe if the
factory-blob model regresses.

Two additional diagnostic patch gates exist for the next stock-app dependency:
`SF2000_PATCH_STOCK_ARCHIVE_SHORTNAME=1` rewrites the app's
`%s/Resources/Archive.sys` format string to a FAT short-name equivalent, and
`SF2000_PATCH_STOCK_ARCHIVE_ACCESS_WAIT=1` bypasses the retry branch after the
same `fs_access()` call. The short-name rewrite does not move the app past the
poll any earlier, which argues against a simple VFAT spelling mismatch. The
access-wait bypass skips the storage-readiness wait and exposes later startup
too early, where the firmware starts audio/video setup and can report
`open font file fail` before the normal remount path catches up. Treat these as
probes for ordering/resource dependencies, not as correctness fixes.

## Display Capture

`make capture-stock-ui` runs the bugfixed stock bootloader against the minimal
generated SD image and writes:

- a QEMU monitor screendump at `build/screenshots/sf2000-stock-capture.ppm`;
- GMA-presented frames under `build/screenshots/gma/` when
  `SF2000_GMA_DUMP_DIR` is enabled by the target.

The generic QEMU screendump can remain black for this headless custom console,
so the GMA frame dump is the more useful comparison artifact. It is written
inside `sf2000_gma_present()` immediately after a descriptor updates the
surface. `sf2000-gma-latest.ppm` is overwritten on every present; numbered
frames preserve the first `GMA_DUMP_LIMIT` presents.

Stock `gma_m36f` descriptors are not always single-framebuffer presents. The
launcher arms a short descriptor chain through word `d6`, then later fills the
backing buffers through GE copy/clear commands. QEMU therefore records the last
active GMA doorbell per display pipe and re-presents those descriptors after a
GE queue completes. Without that continuous-scanout approximation, the menu can
write real pixels into RAM while QEMU keeps showing the older black frame.

RGB565 descriptor chains also include trailing mode-6 blocks whose low bit in
`d0` is set. Stock traces show visible blocks using `...b64` and the following
black/empty tail using `...b65`; treating that low bit as a hidden/end marker
prevents QEMU from compositing the empty tail over the populated menu buffers.
This interpretation is inferred from the stock 08.03 launcher traces and should
be revisited if future captures show a valid visible descriptor with the same
bit set. These hidden tail blocks are quiet in the normal log because they are
expected descriptor-chain structure rather than unknown behavior.

The launcher can also describe a 640x480 RGB565 source buffer with a 320x240
destination rectangle. QEMU scales those source coordinates into the output
rectangle instead of copying the top-left 320x240 pixels. Without this, the
captured menu looks like a zoomed quadrant of the real screen.

## Performance Notes

The SF2000 model should stay accuracy-first, but display paths are hot enough
that small mechanical optimizations matter for iteration speed. RGB565 scanout
uses a 64K host-side lookup table for conversion to QEMU's 32-bit surface
format. This does not change guest-visible timing or memory contents; it only
avoids repeating the same bit expansion for every menu pixel.

GE clear commands write through a fixed-size stack buffer in chunks instead of
allocating a full destination-sized temporary buffer. This keeps large stock
menu surface clears from creating avoidable allocator pressure while preserving
the same guest memory writes.

GE queue and node summaries are off by default. The implemented GE copy/clear
operations are now understood well enough that logging every queue is normal
runtime overhead rather than useful evidence. Set `SF2000_GE_DUMP_LIMIT=<n>`
and optionally `SF2000_GE_NODE_DUMP_LIMIT=<n>` when investigating a new GE
command shape.

Block-backed SDIO DMA reads and writes use a bulk host transfer when the guest
request maps directly onto the raw SD image. The older sector loop remains as
the fallback for synthetic probe media, short/unaligned writes, and host block
errors. This keeps the common stock boot/menu read path faster without hiding
the exact LBA, length, and fallback behavior in the logs.

`SF2000_TRACE_KEYS=1` logs the emulated L23/L24 shift-register samples and
clock transitions. It is intentionally opt-in because the stock `joy_task`
polls frequently once the launcher is running.

`SF2000_TRACE_SDIO=1` includes the otherwise-suppressed SDIO MMIO register
reads/writes plus successful SDIO command and DMA summaries in the QEMU log.
Leave it off for normal boot captures because the stock firmware polls status
registers heavily and reads many sectors during boot; enable it when comparing
real SD-card mode switches or investigating command ordering. Error and
fallback logs remain unconditional.

`SF2000_TRACE_SYSIO=1` re-enables verbose logs for the SYSIO window from
`0x18800000` through `0x1880051c`, including stable clock, reset, interrupt,
GPIO-control, and pinmux setup registers. `SF2000_TRACE_AUX=1` re-enables logs
for the auxiliary command window around `0x1880f000`. Both ranges are quiet by
default because stock display and video-output setup touches them repeatedly,
while the generic register store and explicit GPIO paths still preserve
guest-visible behavior.

Modeled hot peripheral windows such as UART1, SAR ADC, watchdog, timers, GE,
GMA, and AVPHY are also quiet in the generic MMIO logger by default. Their
guest-visible behavior is handled by the device model, so routine register
traffic is not treated as unknown evidence.

The normal trace also suppresses source-identified HC15xx setup windows for
extended SYSCLK registers around physical `0x18801000` and the audio DAC block
around `0x1880b000`. These ranges are used by SDK startup/standby code for
clock and audio-power setup; QEMU keeps their written values in the generic
register store but does not treat routine setup writes as missing devices.
The same is true for the tiny boot-time DSC touch at physical `0x18870000`;
SDK startup clears bit 25 there before the fuller DTS-backed DSC node is
initialized.

`SF2000_TRACE_SFLASH=1` re-enables routine SPI-flash command/read/write logs.
The factory/security-blob behavior is still modelled when this trace is off;
only the high-volume byte stream log is suppressed. `SF2000_TRACE_PC=1` remains
the better option when investigating the security and archive-read landmarks,
because it prints the relevant PC/RA/GP context around those paths.

`make capture-vanilla-ui` downloads the vanilla SF2000 08.03 OS files release,
builds a generated FAT32 SD image with those files, and runs the same capture
flow. This target is optional because it needs `dosfstools`, `mtools`, and
`unzip`, and fetches an external zip into `build/downloads/`; neither the zip
nor the generated SD image is committed.

During launcher bring-up, additional manually generated comparison frames are
kept under directories such as `build/screenshots/fw-trace/` and
`build/screenshots/auto-probe/`. The latest captures from the validation
investigation are still black at the GMA layer; for example:

- `build/screenshots/fw-trace/sf2000-gma-latest.ppm`
- `build/screenshots/auto-probe/sf2000-gma-latest.ppm`
