# SF2000 Technical Notes

These are the non-obvious details currently baked into the QEMU bring-up.

## Memory

The SF2000 has 128 MiB of RAM. The HCRTOS DTS used by the existing ports
defines:

```c
#define CONFIG_MEMORY_SIZE 0x08000000
```

QEMU maps that RAM at physical `0x00000000`. MIPS unmapped cached virtual
addresses at `0x80000000` therefore refer to the same RAM.

## Boot Chain Inputs

There are two stock firmware inputs:

- SPI boot ROM:
  `/root/host-frogdev/universal/orig_firmware/SF2000_XMC_XM25QH40B_4mbit.bin`
- SF2000 application image:
  `/root/host-frogdev/universal/orig_firmware/bisrv_08_03.asd`

The SPI image has reverse-engineered symbols in:

```text
/root/host-frogdev/universal/orig_firmware/SF2000_XMC_XM25QH40B_4mbit.bin.ld
```

Important addresses from that file:

- `lcfg_header = 0x80f00000`
- `image_start = 0x80000200`
- `bl_start = 0x81000000`
- `bl_heap_start = 0x83000000`
- `bl_ramsize = 0x84000000`
- `CHIP_ID = 0xb8800000`

The ASD image starts with an `LCFG` header. The first 0x200 bytes are metadata,
then the image body is mapped as stock firmware data. The stock entry point is
at `0x80001000`.

After the bootloader jumps into `bisrv.asd`, stock maincode calls
`get_sysinfo_from_bl()` at `0x8026e79c`. The bootloader leaves a small handoff
descriptor in low RAM and advertises it through:

- `0xa0000010`: magic `0xabcd5aa5`;
- `0xa0000014`: kseg1 pointer to the handoff payload.

The pointer must be a CPU-visible uncached address, not a physical address.
QEMU seeds the payload at physical `0x00000100` and writes pointer
`0xa0000100`. The first payload word is currently `0x16`, the stock
`RGB_LCD` output route. Writing physical `0x100` into the pointer field makes
the firmware copy routine fault at `0x8028e70c` because it dereferences an
unmapped low virtual address.

Current QEMU supports two modes:

- `-bios <bootrom>`: loads the SPI boot ROM at physical `0x1fc00000`, matching
  the MIPS reset alias at virtual `0xbfc00000`.
- `-bios <bootrom> -kernel <bisrv.asd>`: direct bring-up mode. This loads the
  ASD at physical `0x00000000` and resets the CPU to virtual `0x80001000`.
- `make smoke-stock-full`: boots the stock bootloader with a generated raw SD
  image containing `/BIOS/bisrv.asd`.
- `make smoke-stock-full-bugfix`: boots the vendor bugfixed bootloader against
  the same generated FAT32 image and verifies handoff into the stock firmware.

The direct ASD path is still useful because it reaches the stock application
quickly. The full-chain target currently proves bootloader SD initialization
and raw sector reads from a generated FAT32 image. The stock bootloader reads
the MBR, FAT32 VBR, FSINFO sector, root directory, and `BIOS` directory, then
finds `BISRV.ASD`.

The original bootloader has a known file-object close bug around the directory
scan/open sequence. With minimal synthetic media it can reopen `BISRV.ASD`
through a stale directory pointer and derive compatibility LBA `0x202020` from
space-filled filename bytes. The vendor bugfixed bootloader redirects the
problem call at `0x81003b08` to a wrapper at `0x81020300`; with that image,
QEMU now reads the ASD, passes the bootloader CRC32/MPEG-2 check, enters stock
firmware display code, and mounts the FAT32 SD image.

## Address Translation

Stock firmware accesses registers through MIPS unmapped `kseg1`, for example
`0xb8800000`. QEMU sees those as physical addresses with the `0xa0000000`
offset removed:

```text
0xb8800000 virtual -> 0x18800000 physical
```

That is why the generic MMIO logger reports `0x188...` addresses even though
the firmware symbols use `0xb88...`.

## Chip ID

The first stock ASD block reads `CHIP_ID + 2` and branches on HC15xx values.
Returning zero makes the firmware take the wrong path. QEMU currently returns
`0x3503` for the halfword read at physical `0x18800002`, which matches one of
the known HC15xx branches in the stock startup code.

## Current Boot Status

`make smoke-stock-asd` confirms:

- the SPI image loads at `0x1fc00000`;
- the stock ASD loads at physical `0x00000000`;
- reset starts at `0x80001000`;
- early MMIO reaches the HC15xx chip ID and clock/pinmux setup block.

`make smoke-stock-fatfs` confirms that direct stock ASD execution also reaches
the SD/FatFs mount path. The SDIO model currently implements enough command
response plus DMA read/write behavior for firmware access to a generated card
or an optional raw `sd0` image. It is intentionally permissive: missing media
falls back to synthetic boot sectors so firmware probes keep advancing.

The SDIO DMA register at physical `0x1884c030` is modelled as a start/busy
latch plus completion bits. The stock bootloader writes `0x21` to start, waits
for bits `0x0c`, then acknowledges by writing back bit `0x40`.

The bugfixed stock bootloader path now reaches the stock firmware, mounts the
generated FAT32 image, and exercises GMA scanout. Direct ASD mode remains the
fastest path for display and filesystem iteration.

Implemented peripherals beyond raw boot include:

- GPIO keypad scanning through the active-low L23/L24 shift-register contract.
- Eight south timers with a synthetic scheduler tick on TIMER5/EIRQ3. TIMER0
  and TIMER1 are second/millisecond timers, not generic compare timers; their
  control reads expose current milliseconds in bits 16..25 and must not assert
  `INT_ST` for a zero target.
- SDIO command responses and DMA read/write transfers backed by `sd0`.
- GPIO-driven panel command/pixel writes for early display diagnostics.
- SAR ADC register defaults with a stable battery-range sample.
- Watchdog-disable acknowledgement so firmware can touch WDT without noise.

The early synthetic TIMER5 interrupt is edge-like. Before firmware programs
TIMER5, reading interrupt status bank 1 consumes the synthetic tick and schedules
the next one. Leaving the line level-asserted after that read re-enters the
stock interrupt dispatcher and lands in its nested-interrupt guard at
`0x8033bf38`.

The SDIO completion interrupt shares the same external interrupt path but is
gated by interrupt-enable bank 1. Asserting it while disabled regresses the
bootloader into its interrupt path. With the enable gate in place, the vanilla
08.03 OS image reaches `[FS]successed!` after multi-sector reads from the FAT32
card image.

The next missing pieces are higher-fidelity GE/display timing, audio, USB, and
the rest of the low-power/standby peripheral behavior.

## Display Capture

`make capture-stock-ui` runs the bugfixed stock bootloader against the minimal
generated SD image and writes:

- a QEMU monitor screendump at `build/screenshots/sf2000-stock-capture.ppm`;
- GMA-presented frames under `build/screenshots/gma/` when
  `SF2000_GMA_DUMP_DIR` is enabled by the target.

The generic QEMU screendump can remain black for this headless custom console,
so the GMA frame dump is the more useful comparison artifact. It is written
inside `sf2000_gma_present()` immediately after a descriptor updates the
surface. `sf2000-gma-latest.ppm` is overwritten on every present; numbered
frames preserve the first `GMA_DUMP_LIMIT` presents.

`make capture-vanilla-ui` downloads the vanilla SF2000 08.03 OS files release,
builds a generated FAT32 SD image with those files, and runs the same capture
flow. This target is optional because it needs `dosfstools`, `mtools`, and
`unzip`, and fetches an external zip into `build/downloads/`; neither the zip
nor the generated SD image is committed.
